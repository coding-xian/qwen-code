# 项目架构理解

## CLI 发送请求后，后续流程是什么？

```
用户输入 → CLI (in-process) → Core (SessionService → GeminiClient → LLM)
                                      ↓
                              工具执行、文件操作、MCP 调用
                                      ↓
                              流式响应返回 CLI → 渲染展示
```

CLI 直接作为 Node.js 进程**内部调用** Core 的 API（`SessionService.sendMessage`、`GeminiClient.sendMessageStream`），不走任何中间协议。

---

## 渠道（Telegram/微信等）发送请求后，后续流程是什么？

```
渠道消息 → AcpBridge → [stdin] → qwen --acp (子进程) → Core
                    ← [stdout] ←        (NDJSON 事件流)

渠道收到完整响应 → 调用平台 API 回复用户
```

渠道不直接调 Core，而是通过 ACP 协议与一个独立的 `qwen --acp` 子进程通信（进程隔离），子进程内部调用 Core 处理，结果通过 stdout 的 NDJSON 事件流返回。

---

## CLI 和渠道的本质区别？

| | CLI | 渠道 / serve |
|---|---|---|
| 通信方式 | in-process 直接调 Core API | out-of-process，ACP 协议（JSON-RPC over stdio） |
| 进程模型 | 单进程 | 父进程 + `qwen --acp` 子进程 |
| 适用场景 | 终端交互、脚本调用 | 消息平台集成、Web IDE、远程访问 |

## SDK（Python、Java、TypeScript）是什么作用？

SDK 是**编程接口**，让开发者在代码中调 Qwen Code，不用手动敲命令。

| | 方式 | 典型场景 |
|---|---|---|
| CLI | 人敲命令 | `qwen -m "分析代码"` |
| SDK `query()` | 代码调用，自动起子进程 | `query({ prompt: "分析代码" })` |
| SDK + serve | 代码调用，连远程服务 | `new DaemonClient({ baseUrl: "http://云:4170" })` |

三种 SDK 只是语言不同，作用一样：**让程序能调用 Qwen Code**。

---

## 如果把 `qwen serve` 部署到云端，能看到什么文件？

只能看到**云服务器上的文件**，看不到本地文件。

```
本地电脑                 云端服务器
  file.txt ----看不见--→  qwen serve
                           ↑
                       只能访问服务器磁盘
```

适合的场景就是**代码本来就在云上**：
- CI/CD（GitHub Actions）
- Codespaces / 远程开发机
- 公司内网服务器 + 代码仓库在同一网络

## Core 内部架构是怎样的？

### 核心数据流

```
用户消息 → GeminiClient → GeminiChat → LLM API
                                           ↓
                                    工具调用?
                                     ├── 否 → 返回文本响应
                                     └── 是 → CoreToolScheduler
                                                ├── 校验参数
                                                ├── 权限检查
                                                ├── 执行工具 (ReadFile/Edit/Shell等)
                                                └── 结果回填 → 继续调 LLM
```

### 主要模块

| 模块 | 路径 | 作用 |
|---|---|---|
| **对话入口** | `core/client.ts` | GeminiClient — 管理整个对话周期 |
| **对话循环** | `core/geminiChat.ts` | GeminiChat — 维护历史消息，调 LLM，处理重试和压缩 |
| **LLM 适配层** | `core/contentGenerator.ts` | ContentGenerator 接口，根据 AuthType 选实现 |
| **LLM 实现** | `core/openaiContentGenerator/` | OpenAI 兼容 API |
| | `core/geminiContentGenerator/` | Google Gemini |
| | `core/anthropicContentGenerator/` | Anthropic Claude |
| | `core/qwenContentGenerator.ts` | Qwen OAuth |
| **工具系统** | `tools/` | 所有工具定义（ReadFile、Edit、Shell、Grep 等） |
| | `tools/tool-registry.ts` | 工具注册中心 |
| | `tools/mcp-client-manager.ts` | MCP 服务器管理 |
| | `core/coreToolScheduler.ts` | 工具调度器 |
| **权限** | `permissions/` | PermissionManager、Auto 模式分类器 |
| **配置** | `config/config.ts` | Config 类 — 核心 IoC 容器 |
| **服务** | `services/` | 文件系统、会话管理、压缩、Cron |
| **多 Agent** | `agents/` | Arena（多模型对比）、Team（协作）、Workflow |
| **记忆** | `memory/` | 自动记忆（提取→梦境→回忆→遗忘） |
| **Hook** | `hooks/` | 插件生命周期钩子 |
| **MCP** | `mcp/` | MCP OAuth 认证 |

### Config 是核心枢纽

```
Config (IoC 容器)
  ├── getGeminiClient()     → 对话引擎
  ├── getToolRegistry()     → 工具注册
  ├── getPermissionManager() → 权限
  ├── getMemoryManager()    → 记忆
  ├── getSessionService()   → 会话持久化
  ├── getMcpClientManager() → MCP
  └── getFileSystemService() → 文件操作
```

所有子系统在 Config.initialize() 时创建，通过 getter 访问。

## 静态记忆和动态记忆从哪里加载，如何加载，如何拼接？

### 静态记忆

| 文件 | 路径 | 说明 |
|---|---|---|
| `QWEN.md` | 从 CWD 向上搜每层目录 + `~/.qwen/` | 项目级+全局，用户手动编写 |
| `AGENTS.md` | 同上 | 给 AI 的额外指令，用户手动编写 |
| `QWEN.local.md` | `<项目根>/.qwen/QWEN.local.md` | 开发者个人配置，gitignored |
| 基线规则 | `~/.qwen/rules/*.md` + `<项目>/.qwen/rules/*.md` | 无 `paths:` 字段的规则 |
| 条件规则 | 同上目录 | 有 `paths:` 字段，按匹配到的文件路径按需注入 |

**加载过程** (`memoryDiscovery.ts:loadServerHierarchicalMemory()`)：
1. 从 CWD 向上逐层搜索每个 `include` 目录，找 `QWEN.md`/`AGENTS.md`（`getGeminiMdFilePathsInternal()`）
2. 追加 `<项目根>/.qwen/QWEN.local.md`
3. `readGeminiMdFiles()` 逐个 `fs.readFile()` 读取，处理 `@import` 指令
4. `concatenateInstructions()` 拼接：每段用 `--- Context from: <相对路径> ---` / `--- End of Context from: ... ---` 包裹
5. `loadRules()` 加载规则文件，基线规则追加在静态记忆后面，格式类似 `--- Rule from: ... ---`

### 动态记忆（自动记忆）

| 内容 | 路径 | 说明 |
|---|---|---|
| 项目记忆文件 | `<项目>/.qwen/memory/<type>/<topic>.md` | extract 按 type(user/feedback/project/reference) 分类写入 |
| 项目索引 | `<项目>/.qwen/memory/MEMORY.md` | extract 写完文件后重建 |
| 用户全局记忆文件 | `~/.qwen/memories/<type>/<topic>.md` | 跨项目共享 |
| 用户全局索引 | `~/.qwen/memories/MEMORY.md` | 同上 |

**生命周期**：
- **Extract** (每轮对话后)：fork 子 agent 分析对话历史，写入带 frontmatter 的记忆文件，重建 MEMORY.md 索引
- **Dream** (定期)：fork 子 agent 做合并、去重、修剪、重组
- **Recall** (每轮用户消息时)：扫描项目+用户两套 topic 文件，选相关条目，格式化为 `## Relevant memory` 段落注入当前请求

### 最终拼接顺序（发往 LLM 的 system instruction）

```
1. 基础系统 prompt     (prompts.ts: `You are Qwen Code...`)
2. --- 分隔线
3. QWEN.md + AGENTS.md (memoryDiscovery.ts 拼接结果)
4. 基线规则              (rulesDiscovery.ts: `--- Rule from: ... ---`)
5. --- 分隔线
6. 自动记忆指令+索引区块 (prompt.ts: `# auto memory` + MEMORY.md 索引)
7. Git 状态 (如有)
```

**每轮请求额外注入**：
- 用户消息前：Recall 选中的相关记忆（`## Relevant memory`）作为 system reminder prepend 到请求消息

### 关键拼接点

| 步骤 | 位置 | 做了什么 |
|---|---|---|
| 静态+规则合并 | `memoryDiscovery.ts:520` | `memoryContent + "\n\n" + rulesContent` |
| 静态+动态合并 | `prompt.ts:appendManagedAutoMemoryToUserMemory()` | `静态内容 + "\n\n---\n\n" + 动态记忆区块` |
| 总装 system prompt | `config.ts:refreshHierarchicalMemory()` → `client.ts:getMainSessionSystemInstruction()` → `prompts.ts:getCoreSystemPrompt()` | 基础 prompt + `\n\n---\n\n` + 合并后的用户记忆 |
| 按需注入 recall | `client.ts:2081` | 用户消息时 unshift 到 systemReminders 最前面 |

## 要加记忆，怎么加、加在哪里？

只靠**静态记忆**——你在文件里写什么内容，启动时就会被拼进 system prompt，LLM 每次都能看到。

### QWEN.md（全局—所有项目都生效）

```
位置: C:\Users\just_\.qwen\QWEN.md
```

写你想让 AI 始终知道的偏好、习惯、通用规则。例如：

```markdown
- 回答用中文
- 代码里不要写注释
- 调用 Shell 前先解释命令意图
- 项目根目录是 D:\myfiles
```

### QWEN.md（项目级—只当前项目生效）

```
位置: <项目根>/.qwen/QWEN.md
```

写这个项目特有的约束、架构背景、常用命令：

```markdown
- 此项目是 monorepo，用 npm workspaces
- 包都在 packages/ 下
- 测试用 vitest，要 cd 到对应包目录跑
- 构建命令: npm run build && npm run bundle
```

### AGENTS.md（项目级—给 AI 的额外指令）

```
位置: <项目根>/AGENTS.md
```

和 QWEN.md 加载方式一样，习惯上放项目根目录，语义是给 AI 的补充说明。想用就用，不是必须。

### QWEN.local.md（项目级—不提交 git）

```
位置: <项目根>/.qwen/QWEN.local.md
```

开发者个人配置。已 gitignored，只在你本地生效。适合写个人路径、本地 token 等。

### 规则文件（条件触发）

```
全局: C:\Users\just_\.qwen\rules\*.md
项目: <项目根>/.qwen/rules/*.md
```

不带 `paths:` 字段的规则 = 始终生效；带 `paths:` 字段的规则 = 只在你编辑匹配路径的文件时才生效。

### 小结

| 文件 | 范围 | 生效方式 | 用途 |
|---|---|---|---|
| `~/.qwen/QWEN.md` | 全局 | 每次都加载 | 个人偏好、通用约束 |
| `<项目>/.qwen/QWEN.md` | 项目 | 每次都加载 | 项目架构、约定 |
| `<项目>/AGENTS.md` | 项目 | 每次都加载 | AI 补充指令 |
| `<项目>/.qwen/QWEN.local.md` | 本地 | 每次都加载 | 个人本地配置 |
| `rules/*.md` (无 paths) | 全局/项目 | 每次都加载 | 通用规则 |
| `rules/*.md` (有 paths) | 全局/项目 | 匹配路径时加载 | 文件类型特定规则 |

写完后重启对话（或重载配置）即可生效。**不需要写什么动态记忆**，那是 qwen-code 自己管的事。

## 动态记忆是怎么存储下来的，存在哪里，存成什么样？

### 触发时机

每次你发消息、LLM 回复完成后，如果有新消息，extract 就会触发。

### 写入流程

1. `extract.ts` 检查 extract-cursor.json，看有没有未处理的新消息
2. 如果有 → fork 一个子 agent，给它 `write_file` + `edit` 权限（只能写记忆目录）
3. 子 agent 分析对话历史，按格式写 topic `.md` 文件和更新 MEMORY.md 索引
4. 子 agent 退出后，`indexer.ts` 重新扫描所有 topic 文件，**重新生成** MEMORY.md（覆盖掉 agent 写的临时版本）

### 存在哪里

```
项目级（默认）:
  <runtimeDir>/projects/<sanitizedGitRoot>/memory/
  ├── MEMORY.md               ← 索引
  ├── user/                   ← 你的偏好、习惯
  │   └── role.md
  ├── feedback/               ← 你说的"好/不好/改"
  │   └── testing.md
  ├── project/                ← 项目架构、决策
  │   └── architecture.md
  └── reference/              ← 参考资料
      └── glossary.md

项目级（QWEN_CODE_MEMORY_LOCAL=1 时）:
  <项目根>/.qwen/memory/       ← 同上结构，只是路径在项目内

用户全局（跨项目共享）:
  <runtimeDir>/memories/
  ├── MEMORY.md               ← 同上结构
  ├── user/
  ├── feedback/
  ├── project/
  └── reference/
```

辅助文件在 `memory/` 上一级：
```
<memoryRoot>/../meta.json             ← 元数据（版本、时间戳）
<memoryRoot>/../extract-cursor.json   ← 处理进度（已处理到哪条消息）
<memoryRoot>/../consolidation.lock    ← dream 的锁文件
```

### 存成什么样

**topic 文件**（每个文件一条知识）：

```markdown
---
name: User Memory
description: 用户喜欢简洁回复，减少来回沟通
type: user
---

偏好简短回复，调试时更明显

Why: 用户认为调试场景下简洁回复可以减少来回沟通成本。
How to apply: 回复控制在 3 句话以内，先说答案再说原因。
```

frontmatter 三个字段：`name`（标题）、`description`（一句话描述，recall 用）、`type`（必须是 `user`/`feedback`/`project`/`reference` 之一）。

body 自由写，惯例含 `Why:` 和 `How to apply:` 小节。

**MEMORY.md 索引**：

```markdown
- [User Memory](user/role.md) — 用户喜欢简洁回复
- [Release Freeze](project/release-freeze.md) — 合并冻结开始
- [Testing Policy](feedback/testing.md) — 集成测试用真实数据库
```

每行一条：`- [标题](相对路径) — 描述`。最多 200 行，超过会截断并加警告。

### 后续生命周期（四阶段完整流程）

| 阶段 | 触发时机 | 谁执行 | 详细步骤 |
|---|---|---|---|
| **extract** | 每轮 LLM 回复完成后 | fork 子 agent（`extractionAgentPlanner.ts`） | ① 读 extract-cursor.json，检查是否有未处理的新消息<br>② 如有 → fork 子 agent，权限仅限记忆目录的 write_file + edit<br>③ 子 agent 读对话历史 → 判断哪些值得记住<br>④ 按格式写 topic .md 文件（frontmatter + body）到对应 type 子目录<br>⑤ 更新 MEMORY.md 索引（临时版本）<br>⑥ 子 agent 退出<br>⑦ `indexer.ts:rebuildManagedAutoMemoryIndex()` 重新扫描所有 topic 文件，**从零生成** MEMORY.md（覆盖 agent 写的临时版）<br>⑧ 更新 extract-cursor.json 记录已处理到的位置 |
| **dream** | 定期后台执行（可配置间隔） | fork 子 agent（`dream.ts:runManagedAutoMemoryDream()`） | ① 检查 consolidation.lock，防止并发<br>② fork 子 agent，权限同上<br>③ 子 agent 读所有 topic 文件 + MEMORY.md 索引<br>④ **合并**：多条相似知识合成一条（如 3 条"喜欢简洁"→ 1 条）<br>⑤ **去重**：删除内容完全相同的条目<br>⑥ **修剪**：删掉过时（如已过去的发布日期）、不再相关的内容<br>⑦ **重组**：必要时调整 type 分类或文件拆分/合并<br>⑧ 写回 topic 文件<br>⑨ 子 agent 退出<br>⑩ `indexer.ts` 重新生成 MEMORY.md<br>⑪ 更新 dream cursor / meta.json |
| **recall** | 每轮用户消息发出前（`client.ts:1771`） | 主进程内调用（`recall.ts:resolveRelevantAutoMemoryPromptForQuery()`） | ① `scanAutoMemoryTopicDocuments()` + `scanUserAutoMemoryTopicDocuments()` 扫描项目+用户两套 topic 文件<br>② 解析每个文件的 frontmatter，提取 name + description<br>③ 合并结果：先项目文档后用户文档<br>④ **筛选**：用 LLM 调用或启发式规则，根据当前用户输入挑相关的 topic<br>⑤ 跳过已在本轮注入过的文件（`surfacedRelevantAutoMemoryPaths` 去重）<br>⑥ 格式化为 `## Relevant memory\n...` 段落，标注保存时间和 staleness<br>⑦ 返回给 `client.ts`：用户消息时 **unshift** 到 systemReminders 最前面；ToolResult 轮时 **append** 到请求末尾 |
| **forget** | dream 阶段内隐式执行 | dream 子 agent | ① dream 子 agent 判断某 topic 文件不再有用<br>② 直接删除该 .md 文件<br>③ dream 结束后 `indexer.ts` 重建 MEMORY.md，被删文件自然不在索引中<br>④ 没有独立的"forget 阶段"——forget 是 dream 的一个子动作 |

### 关键点

- 所有 topic 文件都是**子 agent** 写的，不是 extract.ts 直接写
- MEMORY.md 最终由 `indexer.ts` **重新扫描生成**，保证索引和文件一致
- 你**不需要也不建议**手动改 `.qwen/memory/` 下的文件——改了也可能被 extract 覆盖或 dream 合并掉
- 想持久保留某条知识，就写到 QWEN.md（静态记忆），动态记忆是易失的、自动管理的
